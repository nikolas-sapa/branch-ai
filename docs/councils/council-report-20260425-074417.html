What features should Branch ship next?

branch-ai v0.4.0 · solo dev · live on npm · the right next moves, not roadmap-padding
Frame: Branch ships CLI (think/list/export/share/diff), MCP server (4 tools), Next.js viewer with React Flow + click-to-fork/inject + collapsible subtrees, streaming via SSE, semantic diff view, Vercel Blob hosted mode, Yjs real-time presence, and a separate Obsidian plugin. Solo builder, public npm package. What features should ship next that are genuinely valuable — not roadmap-padding?
Chairman's verdict

Fix the floor before raising the ceiling.

The pick
2 weeks: bug fixes (week 1), then retention (week 2)
The product has visible defects (fork dead-end, hidden finalText, blunt truncation) that invalidate every higher-order feature. Ship those eight items, then earn the return visit with search/tagging/provenance. Defer the moonshots until you have evidence users come back.

Week 1 — fix the floor:

  1. Fork dialog dead-end → router.push(/t/<newSessionId>) after fork/inject
  2. Render finalText as a fixed bottom panel in the viewer
  3. Click-to-expand on truncated nodes
  4. Make --stream the default, add --no-stream for legacy

Week 2 — earn the return visit:

  1. branch search "..." and a search bar on viewer home
  2. Tag and pin sessions
  3. Provenance link from any forked node back to the question it forked from
  4. Auto-save partial trees on interruption; flag incomplete sessions visibly

Defer: branch merge, branch replay, branch watch, public reasoning galleries, B2B compliance pivot — real ideas, but castles on a broken foundation today.

Where the council agreed

  • The core action loop is broken. Contrarian and Executor independently named the same defect: clicking fork → filling dialog → seeing nothing change.
  • finalText is invisible. Stored in the JSON, never rendered in the viewer. The thing users actually want to read is hidden.
  • The on-ramp is broken. Outsider flagged that "wraps the Claude CLI" assumes 3 prior pieces of context. Contrarian flagged that v0.4.0 is unshipped while public install gets stale v0.1.2.

All three reviewers independently sided with C: "Concrete, actionable, high-leverage. A broken fork dialog poisons all the positioning work, all the loop rhetoric, and makes D and E irrelevant. Ship working software first."

Where the council clashed

  • Visionary vs. repair. Expansionist proposed branch merge, branch replay, B2B pivot. Reviewers sided with Executor's "fix the four broken things first." Vision without working software is a deck.
  • Yjs presence — theater or premature? Contrarian called real-time presence "multiplayer for a single-player game." Reviewer 3 pushed back: maybe forking is for async review, in which case presence isn't theater, just early. Insufficient data to settle.
  • Onboarding vs. depth. Outsider says fix the funnel before adding features. First Principles says reframe entirely as a "reasoning ledger." Both right at different time horizons.

The blind spot all 5 advisors missed

Do users come back?

All three reviewers raised the same point independently. Every advisor optimized for acquisition (A), scope (B), bug fixes (C), strategic reframe (D), or moonshot (E). None asked about retention. A reasoning tree has zero stickiness if it's a throwaway. Without search, return-to-session UX, and durable persistence through interruption, the "reasoning ledger" and "compliance infrastructure" theses are fiction. The leading indicator isn't installs or forks — it's whether someone opens a past tree and acts on it.

The one thing to do first

Fix the fork dialog dead-end. Two hours of work. After a fork, router.push to the new sessionId instead of refreshing the current page. Until that line of code lands, every other feature is a tree falling in an empty forest — users click, see nothing change, and never return.
